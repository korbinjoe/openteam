UX Proposal · OpenTeam

Make the sidebar more quiet, more useful

When a user manages 100+ parallel tasks, the sidebar is the place they look at 200 times a day. Right now it's too "noisy" — repeated titles, identical badges, recurring "Add Agent" buttons, hover-only tiny icons — everything fighting for attention.

Status: Observed & tested · 8 major issues
Thesis: Upgrade from "list" to "command center"
Round · 1
TL;DR

Three Most Important Changes

  1. 01
    Compress each task from ~80px to 36px. Collapse agent rows and Add Agent by default, expand on hover/active only.
  2. 02
    Add a search + status filter bar. 80% of 218 tasks are historical archives, but there's no filtering mechanism.
  3. 03
    Replace the uniform purple dot with semantic status dots (running / waiting / done / error). The core question of pulse-mode workflow: "While I was away, who moved?"
Why now

OpenTeam's core rhythm is "batch dispatch → leave → batch review."
The sidebar is the first thing seen within 1 second of returning. Currently it takes 5 seconds of scrolling + reading to figure out "who moved, who's stuck, who needs my review." Goal: reduce those 5 seconds to 1 second.

A1 Each task takes ~80px

Title + agent + Add Agent — three persistent rows. 113 tasks occupy 9000+px vertical space; must scroll to reach last week's items.

A2 Many "Generate a concise title…"

When auto-title generation fails, the raw prompt template is shown. Multiple identical titles in a row make the list unreadable.

A3 Pin / Archive only on hover

16×16 tiny icons appear beside an already-truncated title, further compressing title width and triggering layout shift.

A4 Purple dot has no semantics

All tasks show the same purple dot. Can't tell at a glance who's running, who's stuck, who errored — core signal for pulse-mode workflow is missing.

A5 Active highlight on agent row, not task

Current selection state appears on the child agent item, not the parent task. Clicking title vs agent leads to two different destinations — confusing mental model.

A6 No search / filter / grouping

218 tasks listed; finding yesterday's requires manually scrolling 30 screens. No status filter, no today/yesterday grouping.

A7 Collapsing sidebar = losing all tasks

After collapsing with ⌘B, only logo + 4 bottom icons remain. Current task, active tasks all vanish — no way back.

A8 Bottom nav takes ~150px

Team / Skills / Schedules / Workspaces — four rows + 4 icons + theme switcher, all stacked, permanently consuming task list viewport space.

Section 02

Design Principles

P1

Attention First

Quiet by default. Only tasks requiring user decisions "call out" — status changes, review needed, blocked.

P2

Pulse-Friendly

See within 1 second of returning: "3 running / 2 awaiting review / 1 failed." No scrolling required.

P3

Progressive Disclosure

Default 36px minimal per row. Hover reveals actions, active expands agents — never proactively consumes space.

P4

Batch-Native

Super-individuals dispatch 20 tasks and archive 50 tasks daily. Multi-select, batch archive, batch re-dispatch must be first-class citizens.

Grouping & Sorting

Default: status-first → time-second:

  1. Error (most visible, demands action)
  2. Awaiting (blocked on user)
  3. Running (actively executing)
  4. Today · Yesterday · This week · Older (by completion time)

Users can switch to time-only or by-agent mode. Pinned always stays on top.

Implementation Roadmap

Phase 1 · Reduce Noise

  • Collapse task rows by default (hide Add Agent + agent rows)
  • Switch status dots to 4-color semantic (requires server to report status)
  • Use opacity transitions for actions to avoid layout shift

Phase 2 · Add Power

  • Top search + status chips (client-side filtering)
  • Today / Yesterday time grouping
  • Fix title generation fallback (avoid consecutive "Generate a concise title…")

Phase 3 · Advanced

  • Multi-select + floating batch action bar
  • Collapsed-state command center popover (hover Logo trigger)
  • Keyboard navigation J/K/E/P/X
